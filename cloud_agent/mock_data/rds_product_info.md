# 云数据库 RDS 核心架构与多活部署方案

## 1. 产品概述与底层依赖
云数据库 RDS（Relational Database Service）是基于阿里云飞天架构深度优化的数据库服务。它的底层计算节点实际上也是依赖于 ECS 实例（如 ecs.c7.large），存储节点依赖于 ESSD 云盘。

## 2. 主流引擎与适用场景
*   **RDS MySQL**：全球最受欢迎的开源数据库。
    *   **版本支持**：5.7, 8.0。
    *   **适用场景**：高并发交易系统（配合 ecs.g8a.4xlarge 提升并发）、互联网社交、游戏数据存储。
*   **RDS PostgreSQL**：高级开源关系型数据库。
    *   **版本支持**：13, 14, 15。
    *   **适用场景**：GIS 地理信息系统、复杂金融数据处理、推荐系统。

## 3. 部署架构与容灾策略 (同城多活)
RDS 实例为了保证高达 99.99% 的可用性，强烈建议使用企业级高可用架构。

### 3.1 基础版 (Basic)
*   **架构**：单节点部署，无备用节点，底层可能挂载 ESSD PL1。
*   **缺陷**：当底层 ECS 宿主机发生硬件故障时，实例会宕机数分钟直至在另一台物理机上恢复，期间业务中断。主要用于个人博客、开发测试环境，不承诺高可用 SLA。

### 3.2 高可用版 (High Availability)
*   **架构**：一主一备（Master-Standby）双节点架构。主节点处理读写请求，备节点通过内网实时同步数据。
*   **同城容灾部署**：强烈建议将主备节点部署在不同的可用区。
    *   **主节点**：部署在 **华北2（北京）可用区 K (cn-beijing-k)**。
    *   **备节点**：部署在 **华北2（北京）可用区 L (cn-beijing-l)**。
*   **故障转移 (Failover)**：如果可用区 K 发生电力故障导致主节点宕机，RDS 系统会在 30 秒内自动将 VIP（虚拟IP）漂移到可用区 L 的备节点上，业务代码无需修改任何连接串即可恢复服务。

## 4. 读写分离架构
对于 `ecs.c7.8xlarge` 这类高并发 Web 服务器，单一的 RDS 主节点可能成为性能瓶颈。
*   **只读实例 (Read-Only Instance)**：高可用版 RDS 支持挂载最多 10 个只读实例。
*   **读写分离地址**：开通读写分离后，系统提供一个统一的读写分离地址（如 `rm-bp1xxx.mysql.rds.aliyuncs.com`）。Web 应用连接该地址后，写请求自动路由到主节点，读请求根据权重自动分发到各个只读实例。

## 5. ECS 与 RDS 内网互通的 3 大前置条件
这是企业上云最核心的配置，任何一个条件不满足，都会导致 ECS 无法连接 RDS 报 Connection Refused。

1.  **地域与网络相同**：ECS（如 ecs.g8a.xlarge）与 RDS 必须位于**同一个地域（如华北2 北京）**且**同一个专有网络 VPC（如 vpc-prod-beijing）**内。跨 VPC 默认不通，需要通过云企业网（CEN）打通。
2.  **RDS 白名单授权**：RDS 默认拦截所有请求（白名单为空）。**必须**将访问它的 ECS 实例的私网 IP（如 `10.0.0.15`）或者所在的 VPC 网段（如 `10.0.0.0/16`）添加到 RDS 的访问白名单中。
3.  **ECS 出方向安全组放行**：虽然 ECS 的出方向安全组默认是允许所有的，但如果企业设置了严格的出网限制，必须确保放行目标为 RDS 内网 IP、目标端口为 **TCP 3306 (MySQL)** 或 **TCP 5432 (PostgreSQL)** 的流量。
